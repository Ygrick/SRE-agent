    - Что за задача, для кого и какая боль сейчас
    - Что именно сделает PoC на демо
    - Что НЕ делает PoC, явные out-of-scope

    - Обоснование идеи – четкое описание прикладной задачи

    - Потенциальные сценарии использования, в том числе с edge-кейсами
    - Ограничения – предварительная оценка технических (SLO, p95 latency, …) и операционных (бюджет, …) ограничений
    - Архитектурный набросок – краткое описание того, какие модули, интеграции нужны и будут задействованы для решения задачи
    - Потенциальный data flow, какие части могут быть делегированы LLM/Agent, какие нет
- **docs/governance.md:**
    - Riks register – добавить табличку:
        
        *Риск, вероятность/влияние, детект, защита, остаточный риск.*
        
    - Политика логов, работы с персональными данными (если применимо)
    - Защиты от injection, подтверждение действий и т.п.


Зафиксировать архитектуру PoC-системы перед разработкой: состав модулей, их взаимодействие, execution flow, контракты, ограничения и точки контроля. Результат этапа должен быть достаточен, чтобы перейти к реализации без существенных архитектурных пробелов.

- **docs/system-design.md**
    
    Краткий, но полный архитектурный документ, который включает:
    
    - ключевые архитектурные решения;
    - список модулей и их роли;
    - основной workflow выполнения задачи;
    - описание state / memory / context handling;
    - описание retrieval-контура;
    - описание tool/API-интеграций;
    - основные failure modes, fallback и guardrails;
    - Если еще нет: конкретные технические и операционные ограничения: latency, cost, reliability.

- **docs/diagrams/:**
    
    Диаграммы должны объяснять архитектуру, а не дублировать текст.
    
    - C4 Context — система, пользователь, внешние сервисы и границы;
    - C4 Container — frontend/backend, orchestrator, retriever, tool layer, storage, observability;
    - C4 Component — внутреннее устройство ядра системы;
    - Workflow / graph diagram — пошаговое выполнение запроса, включая ветки ошибок;
    - Data flow diagram — как данные проходят через систему, что хранится, что логируется.

- **docs/specs/:**
    
    Для каждого модуля — короткая техническая спецификация, примеры:
    
    - Retriever: источники, индекс, поиск, reranking, ограничения;
    - Tools / APIs: контракты, ошибки, timeout, side effects, защита;
    - Memory / Context: session state, memory policy, context budget;
    - Agent / Orchestrator: шаги, правила переходов, stop condition, retry/fallback;
    - Serving / Config: запуск, конфигурация, секреты, версии моделей;
    - Observability / Evals: какие метрики, логи, трейсы и проверки будут собираться.

**Важный комментарий про треки и проекты:**

1. Если вы на агентском треке, то в дизайне системы в первую очередь делайте упор на взаимодействие с LLM и ее составляющими. Важно проработать качество и его контроль, механики защит и фолбеков.
2. А если вы инфрастуктурном треке, то в первую очередь интересует стабильно и надежность системы. Это включает работу над мониторингами, защитные механизмы при недоступности API с LLM, утилизация и оптимальность использования ресурсов
