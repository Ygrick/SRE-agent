# Product Proposal: AI-SRE Agent

**Обоснование идеи**
Для ускорения обработки инцидентов первой линии технической поддержки требуется интеллектуальная система, способная имитировать действия инженера в терминале Linux. Подход с использованием LLM-агента позволяет отказаться от жестко закодированных скриптов (которые ломаются при изменениях в инфраструктуре) в пользу адаптивного планирования диагностики на лету.

**Цель и метрики**

* **Цель:** Разработать PoC автономного агента, способного без участия человека расследовать типовые инфраструктурные алерты, собирая исчерпывающий контекст через терминал Linux.
* **Продуктовые метрики:** Снижение времени на первичную диагностику инцидента (MTTD) в тестовой среде.
* **Агентские метрики** (evals через LLM-as-a-judge / экспертную оценку):
  * *Полнота собранного контекста:* Доля критически важной информации из терминала, успешно извлеченной агентом (0–100%).
  * *Качество рекомендаций:* Релевантность предложенного решения внутренним Runbook-ам.
  * *Процент галлюцинаций:* Частота выдуманных причин инцидента или несуществующих путей к файлам.
* **Технические метрики:** Успешность завершения цикла (отсутствие бесконечных петель вызова инструментов).



**Потенциальные сценарии использования и Edge-кейсы**

* **Сценарий 1 (Успешный):** Zabbix присылает алерт по RAM. Агент заходит в терминал, делает `docker stats`, видит утечку памяти в контейнере Redis, делает `docker logs --tail 50 redis`, находит ошибку, ищет в Qdrant инструкцию "Redis OOM", пишет отчет в Telegram.
* **Edge-case 1 (Контекстное окно):** Агент решает сделать `cat` огромного лог-файла. *Решение:* Системное ограничение на уровне инструмента bash (принудительное использование `tail`/`head` или обрезка вывода до N символов).
* **Edge-case 2 (Галлюцинация команды):** LLM генерирует синтаксически неверную команду для терминала. *Решение:* Инструмент возвращает `stderr`, агент должен осознать ошибку и переписать команду.

**Ограничения**

* **Технические (Hardware & LLM Serving):** Основная модель — stepfun/step-3.5-flash через OpenRouter (free tier, 2-5s latency). Опционально — vLLM на GPU (gpt-oss-20b, RTX 4090 24GB). LiteLLM Proxy обеспечивает routing и failover между провайдерами.
* **Операционные:** Только опенсорс инструменты. Срок реализации PoC — 8 недель (1 Middle NLP engineer).

**Архитектурный набросок и Data Flow**

* **Модули и интеграции:**
  * *Trigger:* Zabbix Webhook Receiver (Python FastAPI)
  * *Orchestrator/State:* Codex CLI (agent-loop с AGENTS.md инструкциями)
  * *Memory/Knowledge:* Qdrant (векторная БД для Runbooks)
  * *Tools:* Python `subprocess` или SSH-клиент для выполнения bash-команд
  * *Tracing:* Langfuse
  * *Output:* Telegram Bot API


* **Data Flow** (делегирование LLM vs детерминировано):

| Шаг | Исполнитель | Действие |
|-----|-------------|----------|
| 1 | Детерминировано | Zabbix → JSON алерта → FastAPI (webhook receiver) |
| 2 | LLM/Agent | Анализ алерта, формирование плана диагностики |
| 3 | LLM → Детерминировано | Генерация bash-команды → Python-обёртка валидирует и выполняет в терминале |
| 4 | Детерминировано → LLM | stdout/stderr возвращается в контекст агента (Observation). Цикл 2–4 до нахождения причины |
| 5 | LLM → Детерминировано | Формирование отчёта → Отправка в Telegram |

*Не делегируется LLM:* выполнение команд (только валидация и subprocess), приём webhook, отправка в Telegram, поиск в Qdrant (если реализован как отдельный сервис).
